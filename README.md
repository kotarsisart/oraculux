# 🧿 Oraculux
A mystical oracle experience that delivers ambiguous answers, symbolic signals, and illusion of meaning.

---

## 🧠 Concept

Oraculux is designed as a digital oracle — but not in a literal sense.
It does not provide clear answers.
It provides interpretations, signs, and vague guidance — the kind that feels meaningful, but requires the user to project meaning onto it.
Under the surface, this is a subtle irony.
The system mimics:
oracles
fortune tellers
“mystical” systems
But in reality, there is no deep logic behind the answers.
The meaning is constructed entirely by the user.
The project explores how easily humans:
seek patterns
assign meaning
trust ambiguity

---

## 🔮 Experience

The user interacts with a digital oracle interface:
asks a question
receives a stylized, cryptic response
interprets it subjectively
The responses are intentionally:
non-specific
open-ended
emotionally suggestive
This creates an illusion of depth — while remaining structurally simple.

---

## Telegram Integration

From the hero section, users can access an external Telegram bot.
The bot extends the experience:
allows asking questions directly
returns oracle-style answers
mirrors the same ambiguous logic
This bridges the web experience with a more “personal” interaction channel.

---

## 🖱️ Interaction & UX

The project is heavily interaction-driven.
Key ideas:
UI behaves as part of the “mystical system”
feedback is delayed, animated, and intentional
responses feel “generated” rather than displayed
The experience relies on:
anticipation
uncertainty
perception of hidden logic

---

⚙️ Tech Stack

React + TypeScript
SCSS (modular structure)
Custom i18n system
No animation libraries — all animations implemented manually

---

## 🧱 Frontend Complexity

Oraculux is intentionally built as a frontend-heavy system.
The goal was to push interaction and animation to a high level of complexity:
custom animation systems (no external libraries)
fine-grained DOM control where needed
state-driven UI transitions
layered interaction logic across components
In many parts, the frontend behaves more like an interactive engine than a traditional UI.

---

## 🧩 Architecture Highlights

Component-driven architecture
Custom hooks for interaction logic (animations, effects, feedback)
Separation of visual layers and behavioral logic
Reusable UI interaction patterns
Controlled imperative DOM usage for performance-critical effects

## 🎨 Design Philosophy

Ambiguity over clarity
Suggestion over explanation
Emotion over logic
Illusion over truth
The system is not meant to answer.
It is meant to feel like it does.
